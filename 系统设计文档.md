# 智校通——系统设计文档

## 一、系统总体架构

### （一）架构概述

智校通采用前后端分离的三层架构，整体分为**前端交互层**、**应用服务层**和**数据存储层**，并通过外部大模型 API 完成核心 AI 能力调用。

```
┌─────────────────────────────────────────────────────────────────┐
│                         前端交互层                               │
│              Web 端 / 移动端 H5 / 微信小程序                      │
└──────────────────────────┬──────────────────────────────────────┘
                           │ HTTPS / WebSocket
┌──────────────────────────▼──────────────────────────────────────┐
│                         应用服务层                               │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────────────┐ │
│  │  学习辅助模块  │  │ 事务问答模块  │  │   个性化成长助手模块    │ │
│  └──────┬───────┘  └──────┬───────┘  └──────────┬────────────┘ │
│         │                 │                      │              │
│  ┌──────▼─────────────────▼──────────────────────▼───────────┐ │
│  │            核心服务层（RAG 引擎 / 对话管理 / 文件解析）         │ │
│  └──────────────────────────────────────────────────────────┘  │
└──────────┬──────────────────────────┬───────────────────────────┘
           │                          │
┌──────────▼──────────┐   ┌───────────▼────────────────────────┐
│    数据存储层         │   │          外部 AI 服务               │
│  MySQL / PostgreSQL  │   │   大模型 API（OpenAI / 文心 / 通义） │
│  向量数据库（Milvus）  │   │   文档解析 API                     │
│  对象存储（OSS）       │   └───────────────────────────────────┘
└─────────────────────┘
```

### （二）技术选型

| 层次 | 技术方案 | 说明 |
|------|----------|------|
| 前端 | Vue 3 + TypeScript + Vite | 响应式 UI，支持 Web 与 H5 |
| 后端 | Python / FastAPI | 轻量异步框架，便于集成 AI 库 |
| 大模型 API | OpenAI GPT-4o / 百度文心 / 阿里通义 | 可配置，降低单一供应商依赖 |
| RAG 框架 | LangChain / LlamaIndex | 检索增强生成，接入向量数据库 |
| 向量数据库 | Milvus（生产）/ Chroma（开发） | 存储知识库文档 Embedding |
| 关系数据库 | PostgreSQL | 用户数据、对话历史、错题记录等 |
| 文件存储 | MinIO / 阿里云 OSS | 上传的课件、教材文件 |
| 文档解析 | PyMuPDF / python-pptx / python-docx | 解析 PDF、PPT、Word 文档 |
| 缓存 | Redis | 会话缓存、频繁查询结果缓存 |
| 容器化 | Docker + Docker Compose | 本地开发与部署一致性 |

## 二、功能模块设计

### （一）学习辅助模块

#### 1. 文档上传与解析子模块

**流程说明：**
1. 用户上传课件（PDF / PPT / Word）。
2. 后端调用文档解析服务，提取纯文本与结构信息。
3. 对文本进行分块（Chunk），调用 Embedding 模型生成向量。
4. 向量与原始文本块一同写入向量数据库，建立索引。

**关键参数：**
- Chunk 大小：512 Token，重叠 50 Token
- Embedding 模型：`text-embedding-ada-002` 或国产同类模型

#### 2. 知识点提炼子模块

**输入：** 用户选择已解析的文档  
**输出：** 结构化知识点列表（章节 → 知识点 → 简要解释）

**实现方式：** 将文档分块依次送入大模型，使用 Prompt 指引模型输出 Markdown 格式知识点，最终合并去重。

#### 3. 练习题与复习提纲生成子模块

**输入：** 知识点列表 / 指定章节  
**输出：**
- 选择题 / 填空题 / 简答题（可配置题型与数量）
- 复习提纲（大纲结构 + 重点标注）

#### 4. 错题本与复习计划子模块

**数据来源：** 用户手动录入或答题记录自动识别  
**功能：**
- 错题分类存储（按科目、知识点、错误类型）
- 自动生成个性化复习计划（根据错题分布与考试日期）

---

### （二）校园事务 AI 问答模块

#### 1. 知识库管理子模块

**数据来源：** 学校通知、规章制度、办事流程、办事指南等文档  
**处理流程：**
1. 管理员上传或维护知识文档。
2. 按事务类别（请假、奖助学金、证件补办等）分类存储。
3. 向量化后写入 Milvus，支持语义检索。

#### 2. 多轮问答子模块

**流程：**
1. 用户输入问题（如"如何申请助学金？"）。
2. 系统对问题进行 Embedding，在向量数据库中检索 Top-K 相关文档片段。
3. 将检索结果拼接为上下文，通过 RAG Prompt 调用大模型生成回答。
4. 回答附带原文来源与参考链接，支持追问（多轮对话）。

**RAG Prompt 模板：**
```
你是"智校通"校园事务助手，请严格根据以下参考资料回答用户问题。
若资料中无相关信息，请说明无法确认并建议联系相关部门。

参考资料：
{context}

用户问题：{question}
对话历史：{history}
```

#### 3. 模板生成子模块

**功能：** 根据用户填写的基本信息，自动生成标准化申请书、证明材料等文档  
**支持类型：** 请假申请、奖助学金申请、证件补办申请等

---

### （三）个性化成长助手模块

| 功能 | 描述 |
|------|------|
| 学业规划建议 | 基于用户绩点、专业、兴趣生成学期目标与课程建议 |
| 考试计划提醒 | 接入考试日历，生成复习倒计时与阶段计划 |
| 简历优化辅助 | 用户上传简历，AI 给出结构、内容、表达优化建议 |
| 面试表达训练 | 模拟面试问答，AI 给出评分与改进建议 |
| 校园生活信息导航 | 常见问题快速导航（食堂营业时间、图书馆预约等） |

## 六、部署架构

### （一）开发环境

使用 Docker Compose 本地一键启动所有服务：

```yaml
services:
  frontend:     # Vue 3 开发服务器（Vite）
  backend:      # FastAPI 应用
  postgres:     # PostgreSQL 数据库
  redis:        # Redis 缓存
  milvus:       # 向量数据库（含 etcd + minio 依赖）
  minio:        # 本地对象存储（开发用）
```

### （二）生产环境

```
用户
 │
 ▼
Nginx（反向代理 + 静态资源）
 │
 ├──▶ Frontend（Vue 静态构建产物）
 │
 └──▶ Backend（FastAPI，Uvicorn / Gunicorn）
         │
         ├──▶ PostgreSQL（主从）
         ├──▶ Redis（主从）
         ├──▶ Milvus 集群
         └──▶ OSS（阿里云 / 腾讯云对象存储）
```

## 七、安全设计

1. **认证与授权：** 使用 JWT Token + Refresh Token，接口需携带 `Authorization: Bearer <token>`。
2. **文件安全：** 上传文件类型白名单（pdf / pptx / docx / txt），大小限制 50 MB，文件名脱敏存储。
3. **接口限流：** 对 AI 生成类接口设置每用户每分钟请求上限，防止滥用。
4. **数据隔离：** 用户文档与对话数据严格按 `user_id` 隔离，禁止跨用户访问。
5. **Prompt 注入防护：** 对用户输入进行长度限制与特殊字符过滤，避免 Prompt 注入攻击。
6. **HTTPS：** 生产环境全站 HTTPS，禁止 HTTP 访问。
